# Video Analysis

Produce an HTML report from a video: keyframes (with timestamps) each paired with concise
bullet-point takeaways, generated by transcribing the audio and analyzing every keyframe
against the transcript with a vision model.

## Prerequisites

- **ffmpeg + ffprobe** on PATH (`which ffmpeg ffprobe`). On macOS: `brew install ffmpeg`.
- **OpenAI key** in `OPENAI_KEY` or `OPENAI_API_KEY` (the script checks both). The key needs
  both audio-transcription scope and model/responses scope.
- Python deps are pulled at runtime via `uv run --with openai` — nothing to install.

## Usage

One command runs the whole pipeline:

```bash
uv run --with openai python scripts/analyze_video.py INPUT.mp4 --topic "Acme's product"
```

Output lands in `outputs/INPUT_analysis/` inside this skill's own folder (override with
`--out DIR`): `audio.mp3`, `transcript.json`, `keyframes/*.jpg`, `analysis/*.json`,
`report.html`. Open `report.html` in a browser. `outputs/` is gitignored, so reports
never get checked into this repo.

**Always pass `--topic`** with the subject of the video (e.g. `"Sierra's product"`,
`"the Q3 earnings call"`). It sharpens the takeaway prompt; without it the model uses a
generic prompt.

## How it works (6 cached stages)

1. Extract audio (ffmpeg → mono/16k/32kbps mp3).
2. Transcribe with `whisper-1` (`verbose_json` → timestamped segments). Auto-splits audio
   into ~90-min chunks if it would exceed Whisper's 25 MB limit, offsetting timestamps.
3. Select keyframe timestamps: ffmpeg **scene-change detection + a fixed-interval coverage
   floor**, then dedupe. (Why both — see Tuning.)
4. Extract one JPEG per timestamp; the timestamp is in the filename
   (`keyframe_0222_141.57s.jpg` = 02:22 / 141.57s).
5. For each keyframe, send the image + the transcript spoken in `[t-8s, t+20s]` to the vision
   model → JSON `{relevant, title, bullets}`. Frames the model marks `relevant: false` (no
   content tied to `--topic`, e.g. intro small talk, blank frames) are dropped from the report.
6. Build a self-contained `report.html` (screenshots base64-embedded, chronological, with
   collapsible transcript context per frame) from the remaining frames.

**Every stage caches to the output dir**, so reruns skip finished work and only new/failed
frames get re-analyzed. Pass `--force` to recompute everything. This makes it cheap to rerun
after tuning knobs or if a run is interrupted.

## Tuning (common knobs)

- `--scene-threshold` (default `0.05`): lower catches subtler changes (more frames), higher
  keeps only hard cuts. **Low-motion footage** (talking-head, screen-share) has almost no hard
  cuts — scene detection alone under-samples, which is exactly why the `--interval` floor
  exists to guarantee even coverage.
- `--interval` (default `75` seconds): the coverage floor. Set `0` to use scene cuts only.
- `--min-gap` (default `12`): minimum seconds between kept frames (dedupe).
- `--max-frames` (default `0` = no cap): cap total frames (evenly subsampled) to control
  vision-model cost. Cost ≈ one vision call per kept frame.
- `--model` (default `gpt-5.5`): vision model. `--transcribe-model` (default `whisper-1`).

To calibrate the scene threshold for a specific video, inspect the score distribution:

```bash
ffmpeg -i INPUT.mp4 -vf "select='gt(scene,0.01)',metadata=print:file=-" -an -f null - 2>&1 \
  | grep -oE 'lavfi.scene_score=[0-9.]+'
```

## Gotchas (learned the hard way)

- **`OPENAI_KEY` vs `OPENAI_API_KEY`**: the OpenAI SDK defaults to `OPENAI_API_KEY`, but the
  script reads `OPENAI_KEY` too and passes the key explicitly.
- **Too few frames** for a long video → footage is low-motion; lower `--scene-threshold` or
  lower `--interval` (e.g. `45`).
- **Model output not valid JSON**: the prompt demands a JSON object and the script salvages a
  `{...}` substring; a frame that still fails is cached as `(unparsed response)` — delete that
  one analysis JSON and rerun to retry just it.
